당신은 프로젝트의 오케스트레이터입니다. 사용자의 컨텍스트를 분석하여 필요한 작업을 식별하고, Agent 도구를 사용하여 병렬로 실행하세요.

## 사용자 컨텍스트
$ARGUMENTS

## 실행 절차

### Phase 1: 분석 및 계획 (직접 수행)
1. 사용자 컨텍스트를 읽고 필요한 작업을 분류하세요:
   - **DB 변경**: 새 테이블/컬럼 필요 → generate-migration 패턴
   - **백엔드 스택**: 새 Service/Repository/API → new-service 패턴
   - **이벤트 시스템**: 도메인 이벤트 추가 → add-event 패턴
   - **프론트엔드**: 새 컴포넌트 필요 → create-component 패턴
   - **테스트**: 생성된 코드 테스트 → test-for 패턴

2. 작업 간 의존성을 파악하세요:
   - DB 마이그레이션 → Repository → Service → API Route (순차 의존)
   - 컴포넌트 ↔ 백엔드 (병렬 가능)
   - 이벤트 정의 → 이벤트 발행 (순차)
   - 테스트는 항상 마지막

3. TaskCreate 도구로 각 작업을 등록하세요.

### Phase 2: 탐색 에이전트 (병렬 실행)
Agent 도구를 사용하여 Explore 타입 에이전트를 병렬로 실행하세요:
- **에이전트 A**: 관련 기존 코드 탐색 (유사 서비스, 컴포넌트, 타입 확인)
- **에이전트 B**: DB 스키마 및 마이그레이션 현황 탐색
- **에이전트 C**: 기존 이벤트 시스템 및 테스트 패턴 탐색

### Phase 3: 구현 에이전트 (의존성 고려하여 실행)

**3a. 독립 작업 — 병렬 실행** (Agent 도구, 단일 메시지에 여러 호출):
각 에이전트에게 해당 스킬의 전체 지침을 프롬프트에 포함하세요.

예시 병렬 그룹:
- DB 마이그레이션 생성 에이전트
- 타입 정의 에이전트
- 프론트엔드 컴포넌트 에이전트 (백엔드 API 스펙만 합의 후)

**3b. 의존 작업 — 순차 실행**:
- Repository (마이그레이션 결과 참조)
- Service (Repository 참조)
- API Route (Service 참조)
- 이벤트 연결 (Service에 이벤트 emit 추가)

**3c. 최종 단계 — 테스트 및 검증**:
- 생성된 모든 코드에 대한 테스트 생성
- `pnpm lint && pnpm type-check && pnpm test` 실행

### Phase 4: 통합 검증 (직접 수행)
1. 모든 에이전트 결과를 수집
2. 파일 간 import/export 일관성 확인
3. 타입 호환성 확인
4. 누락된 연결 보완
5. 최종 `pnpm build` 실행

### Phase 5: 서빙 파이프라인 검증 (필수 — 생략 금지)
이 서비스는 다수 사용자가 이용 중입니다.
- 변경된 코드가 영향을 주는 모든 요청 경로를 미들웨어 → route handler → 브라우저까지 추적
- CSP 헤더가 이중 적용되는 경로가 없는지 확인 (middleware.ts + 각 route handler)
- assembleHtml() 함수가 CSS/JS를 누락 없이 주입하는지 확인
- "A 경로에서는 되지만 B 경로에서는 안 된다" 같은 차이가 없는지 확인
- ❌ 항목이 있으면 커밋하지 마세요

## 에이전트 프롬프트 작성 규칙
각 에이전트에게 보내는 프롬프트에는 반드시 포함:
1. **프로젝트 컨텍스트**: 기술 스택, 아키텍처 레이어 설명
2. **구체적 작업 지시**: 해당 스킬(.claude/commands/ 내 파일)의 전체 지침
3. **참조 파일 경로**: 기존 패턴을 따를 참고 파일들
4. **출력 기대치**: 생성할 파일 경로와 내용의 구체적 설명
