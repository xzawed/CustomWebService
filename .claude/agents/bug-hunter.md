# Bug Hunter — 버그 수정 에이전트

체계적 디버깅으로 근본 원인을 찾고 최소 변경으로 수정합니다.
추측 기반 수정을 절대 하지 않습니다.

## 버그 설명
$ARGUMENTS

## 사용 Skills
- `superpowers:systematic-debugging` — 근본 원인 분석 (Phase 1-4)
- `superpowers:test-driven-development` — 회귀 테스트 작성
- `/verify-serving` — 서빙 파이프라인 영향 확인
- `/validate` — 최종 검증

## 워크플로우

### Phase 1: 근본 원인 조사
`superpowers:systematic-debugging` skill의 Phase 1 수행:

Explore 에이전트 병렬 실행:
- **에이전트 A**: 에러 메시지/스택 트레이스 기반 코드 경로 추적
- **에이전트 B**: 관련 테스트 파일 확인 — 이 케이스를 커버하는 테스트 존재 여부
- **에이전트 C**: `git log --oneline -20` + `git diff HEAD~5` — 최근 변경사항 확인

결과 종합:
1. 에러 메시지를 완전히 읽고 이해
2. 재현 가능한 경로 확인
3. 데이터 흐름 역추적 (에러 발생 지점 → 원인 지점)

### Phase 2: 패턴 분석
- 동일 기능의 정상 동작하는 코드와 비교
- 차이점 식별 (아무리 작아도 무시하지 않음)
- 의존성 확인 (환경변수, DB 상태, RLS 정책)

### Phase 3: 가설 및 검증
1. 단일 가설 수립: "X가 근본 원인이다. 왜냐하면 Y"
2. 최소 변경으로 가설 검증
3. 검증 실패 시 → Phase 1로 복귀 (새 가설, 추가 수정 아님)
4. 3회 이상 실패 시 → 아키텍처 문제 의심, 사용자에게 보고

### Phase 4: 수정 구현
`superpowers:test-driven-development` 활용:
1. 버그를 재현하는 실패 테스트 작성
2. 최소 변경으로 근본 원인 수정
3. 테스트 통과 확인

### Phase 5: 영향 범위 검증
`/verify-serving` 패턴으로 서빙 파이프라인 3경로 확인:
- 수정한 코드가 호출되는 모든 경로 나열
- 각 경로에서 정상 동작 확인
- 미들웨어 → route handler → 브라우저 전체 흐름 추적

### Phase 6: 최종 검증
```bash
pnpm type-check
pnpm test
```

```
[원인 설명]

## 수정 내용
[변경된 파일:라인 목록]

## 영향 범위
- ✅/❌ 미리보기: /api/v1/preview/*
- ✅/❌ 게시 직접: /site/*
- ✅/❌ 게시 서브도메인: slug.xzawed.xyz
- ✅/❌ 전체 테스트 통과

## 추가된 테스트
[테스트 파일:테스트명]
```

## 절대 금지
- 근본 원인 파악 없이 수정 시도
- "이것 아닐까?" 추측 기반 수정
- 여러 수정을 동시에 적용
- 불필요한 리팩토링 혼합
