## 작업 1: DB에 54개 API 데이터 넣기 (3분)

> 현재 DB에는 이전 15개 API만 들어있습니다.
> 새로 추가한 54개 API(국내 15개 + 해외 39개)를 넣어야 합니다.

### 순서

**1-1.** 웹 브라우저에서 **Supabase 대시보드**를 엽니다.
```
https://supabase.com/dashboard
```
아직 로그인이 안 되어 있으면 로그인하세요.

**1-2.** 왼쪽 목록에서 **본인의 프로젝트**(custom-web-service)를 클릭합니다.

**1-3.** 왼쪽 메뉴에서 **SQL Editor** 를 클릭합니다.
- 아이콘이 `</>` 모양으로 생겼습니다
- 왼쪽 사이드바 중간쯤에 있습니다

**1-4.** 화면 중앙에 빈 편집창이 나타납니다. 여기에 아래 내용을 **복사해서 붙여넣기**하세요.

GitHub에서 최신 seed.sql 파일을 열어 복사합니다:
```
https://github.com/xzawed/CustomWebService/blob/master/supabase/seed.sql
```

또는 로컬 컴퓨터에서 이 파일을 메모장으로 열어 전체 복사합니다:
```
D:\Source\CustomWebService\supabase\seed.sql
```

**1-5.** 복사한 내용을 SQL Editor 편집창에 붙여넣기 합니다.

**1-6.** 편집창 오른쪽 아래의 **"Run"** 버튼(초록색)을 클릭합니다.

**1-7.** 하단에 **"Success"** 메시지가 나오면 성공입니다.
- "54개의 무료 API 정보와 7개의 기능 설정값이 입력되었습니다"
- 만약 에러가 나오면 아래 "문제 해결" 섹션을 참고하세요

**1-8.** 확인하기: 왼쪽 메뉴에서 **Table Editor**를 클릭합니다.
- 상단 테이블 목록에서 **api_catalog**을 선택합니다
- 행(row)이 **54개** 있으면 정상입니다
- 한국어 설명(description)이 보이면 정상입니다

### 문제가 생겼을 때

| 에러 메시지 | 원인 | 해결 방법 |
|------------|------|----------|
| `duplicate key value violates unique constraint` | 이미 같은 데이터가 있음 | 정상입니다. seed.sql에 `ON CONFLICT DO NOTHING`이 포함되어 있어 자동 처리됩니다 |
| `permission denied` | 권한 부족 | Supabase 대시보드에 프로젝트 소유자 계정으로 로그인되어 있는지 확인하세요 |
| `syntax error` | SQL 복사가 잘못됨 | seed.sql 파일을 처음부터 끝까지 다시 복사해서 붙여넣으세요 |

## 작업 2: 서브도메인 DNS 설정하기 — Cloudflare (10분)

> `xzawed.xyz` 도메인으로 서비스에 접속할 수 있게 하고,
> `xxx.xzawed.xyz` 형태로 사용자들이 만든 서비스를 게시할 수 있게 합니다.

### 사전 확인

- `xzawed.xyz` 도메인을 구매했고, Cloudflare에서 관리하고 있어야 합니다
- Cloudflare 계정에 로그인할 수 있어야 합니다
- 아직 도메인을 구매하지 않았다면 이 단계는 건너뛰세요 (Railway 기본 URL로 계속 사용 가능)

### 순서

**2-1.** 웹 브라우저에서 **Cloudflare 대시보드**를 엽니다.
```
https://dash.cloudflare.com
```

**2-2.** 로그인 후, 도메인 목록에서 **xzawed.xyz**을 클릭합니다.

**2-3.** 왼쪽 메뉴에서 **DNS** → **Records**를 클릭합니다.

**2-4.** **"Add record"** 버튼(파란색)을 클릭합니다.

**2-5.** 첫 번째 레코드를 추가합니다 (메인 도메인):

| 항목 | 입력할 값 |
|------|----------|
| Type | `CNAME` (드롭다운에서 선택) |
| Name | `@` (골뱅이 기호) |
| Target | `customwebservice-production.up.railway.app` |
| Proxy status | **켜기** (주황색 구름 아이콘이 되어야 합니다) |
| TTL | `Auto` (기본값 그대로) |

**"Save"** 버튼을 클릭합니다.

**2-6.** 다시 **"Add record"** 버튼을 클릭하여 두 번째 레코드를 추가합니다 (서브도메인):

| 항목 | 입력할 값 |
|------|----------|
| Type | `CNAME` |
| Name | `*` (별표) |
| Target | `customwebservice-production.up.railway.app` |
| Proxy status | **켜기** (주황색 구름) |
| TTL | `Auto` |

**"Save"** 버튼을 클릭합니다.

**2-7.** DNS Records 목록에 다음 두 줄이 보이면 성공입니다:

```
CNAME    @    customwebservice-production.up.railway.app    Proxied
CNAME    *    customwebservice-production.up.railway.app    Proxied
```

**2-8.** (선택) SSL 설정 확인:
- 왼쪽 메뉴에서 **SSL/TLS** → **Overview**를 클릭합니다
- 암호화 모드가 **"Full (strict)"**로 되어 있는지 확인합니다
- 아니라면 **"Full (strict)"**로 변경하세요

### 문제가 생겼을 때

| 상황 | 해결 방법 |
|------|----------|
| CNAME에 `@`를 입력할 수 없다 | Cloudflare는 `@` 대신 도메인 이름(xzawed.xyz)을 직접 입력해도 됩니다. 또는 Name 칸을 비워두면 자동으로 루트 도메인이 됩니다 |
| "CNAME flattening" 경고가 나온다 | 정상입니다. Cloudflare는 루트 도메인의 CNAME을 자동으로 처리합니다 |
| 이미 A 레코드가 있다 | 기존 A 레코드를 삭제하고 위의 CNAME을 추가하세요 |

> DNS 변경사항이 전 세계에 퍼지는 데 최대 24시간 걸릴 수 있습니다.
> 보통 Cloudflare는 5분 이내에 반영됩니다.

## 작업 3: Railway에 도메인 연결하기 (5분)

> Railway에게 "이 도메인으로 오는 요청은 우리 서비스로 보내줘"라고 알려주는 작업입니다.

### 순서

**3-1.** 웹 브라우저에서 **Railway 대시보드**를 엽니다.
```
https://railway.app/dashboard
```

**3-2.** 프로젝트 목록에서 **CustomWebService** 프로젝트를 클릭합니다.

**3-3.** 화면에 보이는 서비스 블록(보라색/파란색 박스)을 클릭합니다.
- 보통 `customwebservice` 또는 `web` 같은 이름입니다

**3-4.** 오른쪽에 나타나는 패널에서 **"Settings"** 탭을 클릭합니다.

**3-5.** 아래로 스크롤하여 **"Networking"** 섹션을 찾습니다.

**3-6.** **"Custom Domain"** 영역에서 **"+ Add Domain"** 버튼을 클릭합니다.

**3-7.** 입력창에 첫 번째 도메인을 입력합니다:
```
xzawed.xyz
```
**Enter**를 누르거나 **"Add"**를 클릭합니다.

**3-8.** 다시 **"+ Add Domain"**을 클릭하고 두 번째 도메인을 입력합니다:
```
*.xzawed.xyz
```
**Enter**를 누르거나 **"Add"**를 클릭합니다.

**3-9.** 각 도메인 옆에 상태가 표시됩니다:
- **"Pending"** → DNS가 아직 전파 중 (몇 분 기다리세요)
- **"Active"** / 초록색 체크 → 연결 성공!
- **"SSL Certificate Issued"** → HTTPS 인증서 발급 완료

**3-10.** 5~10분 기다린 후 페이지를 새로고침(F5)하면 상태가 업데이트됩니다.

### 문제가 생겼을 때

| 상황 | 해결 방법 |
|------|----------|
| 계속 "Pending" 상태 | DNS가 아직 전파되지 않은 것. 최대 1시간 기다려보세요 |
| "Domain already in use" | 다른 Railway 프로젝트에서 이미 사용 중. 해당 프로젝트에서 도메인을 제거하세요 |
| 와일드카드(`*`) 도메인 추가가 안 된다 | Railway 유료 플랜이 필요할 수 있습니다. Railway의 Pro 플랜을 확인하세요 |

## 작업 4: Railway 환경변수 추가하기 (3분)

> 서비스가 "내 도메인이 xzawed.xyz이야"라는 것을 알 수 있게 설정합니다.

### 순서

**4-1.** 작업 3에서 열어둔 Railway 서비스 페널에서 **"Variables"** 탭을 클릭합니다.
- Settings 탭 왼쪽에 있습니다

**4-2.** 기존 환경변수 목록이 보입니다. 여기에 새 변수를 추가합니다.

**4-3.** **"+ New Variable"** 버튼을 클릭합니다.

**4-4.** 왼쪽 칸(이름)에 입력:
```
NEXT_PUBLIC_ROOT_DOMAIN
```

**4-5.** 오른쪽 칸(값)에 입력:
```
xzawed.xyz
```

**4-6.** **Enter**를 누르거나 체크 아이콘을 클릭합니다.

**4-7.** 화면 아래쪽에 **"Deploy"** 또는 **"Redeploy"** 버튼이 나타나면 클릭합니다.
- 환경변수 변경 후 서비스를 재시작해야 반영됩니다
- 재배포는 보통 1~3분 걸립니다

**4-8.** 배포가 완료되면 **"Deployments"** 탭에서 최신 배포가 초록색 **"Active"** 상태인지 확인합니다.

## 작업 5: Supabase 로그인 URL 업데이트 (5분)

> Google/GitHub 로그인 후 사용자가 돌아올 주소를 새 도메인으로 바꿔줍니다.
> 이걸 안 하면 로그인 후 "이전 주소로 돌아가려다 에러가 나는" 문제가 생깁니다.

### 순서

**5-1.** 웹 브라우저에서 **Supabase 대시보드**를 엽니다.
```
https://supabase.com/dashboard
```

**5-2.** 프로젝트(custom-web-service)를 클릭합니다.

**5-3.** 왼쪽 메뉴에서 **Authentication** (자물쇠 아이콘)을 클릭합니다.

**5-4.** 상단 탭에서 **"URL Configuration"**을 클릭합니다.
- "Providers" 탭 옆에 있습니다

**5-5.** **Site URL** 칸을 찾습니다. 현재 값을 다음으로 변경합니다:
```
https://xzawed.xyz
```
- 기존에 `http://localhost:3000`이 적혀 있었을 수 있습니다
- 반드시 `https://`로 시작해야 합니다 (http가 아님!)

**5-6.** **"Save"** 버튼을 클릭합니다.

**5-7.** 아래로 스크롤하여 **"Redirect URLs"** 섹션을 찾습니다.

**5-8.** **"Add URL"** 버튼을 클릭하고 다음 URL들을 **하나씩** 추가합니다:

```
https://xzawed.xyz/callback
```
(Add 클릭)

```
https://customwebservice-production.up.railway.app/callback
```
(Add 클릭)

**5-9.** 기존에 있는 URL들은 **삭제하지 마세요**. 특히 다음은 남겨두세요:
```
http://localhost:3000/callback
```
(로컬 개발할 때 필요합니다)

**5-10.** 최종적으로 Redirect URLs 목록에 다음이 모두 있어야 합니다:

```
http://localhost:3000/callback          ← 기존 (로컬 개발용)
https://xzawed.xyz/callback   ← 새로 추가 (운영용)
https://customwebservice-production.up.railway.app/callback  ← 새로 추가 (Railway 직접 접속용)
```

## 작업 6: 모든 게 정상인지 확인하기 (5분)

> 위 작업들이 모두 끝났으면, 실제로 잘 되는지 확인합니다.

### 확인 1: 메인 사이트 접속

**6-1.** 웹 브라우저를 열고 주소창에 입력합니다:
```
https://xzawed.xyz
```

**6-2.** 확인할 것:
- [ ] 랜딩 페이지가 보인다 (다크 배경, "코딩 없이 만드는 나만의 웹서비스" 텍스트)
- [ ] 주소창에 자물쇠 아이콘이 있다 (HTTPS 정상)
- [ ] "시작하기" 버튼을 클릭하면 로그인 페이지로 이동한다

### 확인 2: API 카탈로그

**6-3.** 상단 메뉴에서 **"카탈로그"**를 클릭하거나 주소창에 입력합니다:
```
https://xzawed.xyz/catalog
```

**6-4.** 확인할 것:
- [ ] API 카드들이 보인다
- [ ] 카테고리 탭(날씨, 금융, 교통 등)이 보인다
- [ ] 검색창에 "날씨"를 입력하면 관련 API만 표시된다
- [ ] API 카드의 설명이 **한국어**로 되어 있다
- [ ] 아무 API를 클릭하면 상세 모달이 **에러 없이** 열린다

### 확인 3: 로그인

**6-5.** 상단 오른쪽의 **"로그인"** 버튼을 클릭합니다.

**6-6.** **"Google로 계속하기"** 또는 **"GitHub로 계속하기"** 버튼을 클릭합니다.

**6-7.** 확인할 것:
- [ ] Google/GitHub 로그인 화면이 나온다
- [ ] 로그인 후 대시보드(`/dashboard`)로 자동 이동한다
- [ ] 상단 오른쪽에 본인 프로필 사진(또는 아이콘)이 보인다

### 확인 4: 서브도메인 (사용자 서비스 게시)

**6-8.** 만약 이전에 게시한 서비스가 있다면, 해당 서브도메인으로 접속해봅니다:
```
https://[slug].xzawed.xyz
```
(slug는 프로젝트 게시 시 생성된 주소입니다)

**6-9.** 확인할 것:
- [ ] 게시된 웹서비스 HTML이 보인다
- [ ] 없는 slug를 입력하면 404 페이지가 보인다

### 확인 5: 법적 페이지

**6-10.** 페이지 하단(Footer)의 링크를 클릭합니다:
- [ ] "이용약관" → `/terms` 페이지가 보인다
- [ ] "개인정보처리방침" → `/privacy` 페이지가 보인다
- [ ] "면책 조항" → `/disclaimer` 페이지가 보인다

## 자주 묻는 질문 (FAQ)

### Q: 도메인을 아직 구매하지 않았어요.
**A:** 도메인 없이도 서비스는 동작합니다. Railway가 제공하는 기본 URL인
`https://customwebservice-production.up.railway.app`으로 계속 사용 가능합니다.
다만 서브도메인 게시(`xxx.xzawed.xyz`) 기능은 사용할 수 없습니다.
작업 2, 3, 4를 건너뛰시면 됩니다.

### Q: Cloudflare 대신 다른 DNS 서비스를 사용하고 있어요.
**A:** 원리는 같습니다. DNS 관리 페이지에서 CNAME 레코드 2개를 추가하면 됩니다:
- `xzawed.xyz` → `customwebservice-production.up.railway.app`
- `*.xzawed.xyz` → `customwebservice-production.up.railway.app`
SSL 인증서는 Railway가 자동 발급합니다.

### Q: 작업 2를 했는데 사이트가 안 열려요.
**A:** DNS 전파에 시간이 걸릴 수 있습니다.
1. 5~10분 기다린 후 다시 시도하세요
2. 브라우저 캐시를 지우세요 (Ctrl+Shift+Delete → "캐시된 이미지 및 파일" 선택 → 삭제)
3. 다른 브라우저나 시크릿 모드로 시도하세요
4. 24시간이 지나도 안 되면 DNS 설정을 다시 확인하세요

### Q: 로그인은 되는데 대시보드로 이동하지 않아요.
**A:** 작업 5의 Redirect URL 설정을 확인하세요.
`https://xzawed.xyz/callback` 이 Redirect URLs 목록에 있어야 합니다.

### Q: 카탈로그에 API가 15개만 보여요.
**A:** 작업 1을 아직 수행하지 않은 것입니다. seed.sql을 Supabase SQL Editor에서 실행하세요.

### Q: seed.sql 실행했는데 여전히 15개만 보여요.
**A:** 브라우저를 새로고침(F5)하세요. 그래도 안 되면:
1. Supabase Table Editor → api_catalog 테이블 확인
2. 행 수가 54개인지 확인
3. 54개가 맞다면 웹사이트 캐시 문제이므로 Ctrl+Shift+R (강력 새로고침)

### Q: 나중에 API를 더 추가하고 싶어요.
**A:** Supabase Table Editor → api_catalog 테이블에서 직접 행을 추가할 수 있습니다.
또는 개발자에게 seed.sql 업데이트를 요청하세요.

### Q: 이 작업들을 개발 서버(localhost)에서도 해야 하나요?
**A:** 아닙니다. 작업 2~5는 **운영 서버(인터넷에 공개된 사이트)** 전용입니다.
로컬 개발 환경은 이미 설정되어 있으므로 `pnpm dev`로 바로 사용 가능합니다.
작업 1(seed.sql)만 운영 DB에서 실행하면 됩니다.
