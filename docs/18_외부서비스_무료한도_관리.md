# 외부 서비스 무료 한도 관리 (Free Tier Limit Management)

> 모든 인프라를 무료로 운영하기 위한 한도 관리 전략
> 한도 초과 시 서비스 장애로 직결되므로 사전 관리가 핵심

---

## 1. 서비스별 무료 한도 상세

### 1.1 Vercel (플랫폼 호스팅)

| 항목 | 무료 한도 | 주의사항 |
|------|-----------|---------|
| 대역폭 | 100 GB / 월 | 이미지 무거운 페이지 주의 |
| Serverless 함수 실행 | 100 GB-hours / 월 | 코드 생성 API가 주 소비 |
| Serverless 함수 시간 | 10초 (기본), 최대 60초 | 코드 생성 시 SSE 스트리밍으로 대응 |
| 빌드 | 100시간 / 월 | PR별 Preview 빌드 포함 |
| 프로젝트 수 | 200개 | 생성 서비스 배포 시 프로젝트 증가 |
| 팀 멤버 | 1명 (Hobby) | 개인 계정만 |
| 분석 | 2,500 이벤트 / 월 | 기본 분석만 |

**위험 시나리오:**
- 생성 서비스 200개 도달 시 신규 배포 불가
- SSE 스트리밍이 60초 초과 시 타임아웃

**대응 방안:**
```
프로젝트 수 관리:
├── 비활성 서비스 자동 정리 (90일 미사용)
├── 사용자당 최대 서비스 5개 제한
└── 한도 도달 시 GitHub Pages로 대체 배포

Serverless 시간 관리:
├── 코드 생성 로직 최적화 (프롬프트 간결화)
├── 생성 결과 캐싱 (유사 요청 재사용)
└── 비피크 시간대 생성 유도
```

### 1.2 Supabase (데이터베이스)

| 항목 | 무료 한도 | 주의사항 |
|------|-----------|---------|
| 데이터베이스 | 500 MB | generated_codes 테이블이 주 소비 |
| 대역폭 | 5 GB / 월 | API 호출 빈도에 비례 |
| Storage | 1 GB | 현재 미사용 예정 |
| Edge Functions | 500,000 호출 / 월 | 현재 미사용 예정 |
| Auth MAU | 50,000 | 충분 |
| Realtime | 200 동시 접속 | 충분 |
| 일시정지 | 7일 비활성 시 자동 일시정지 | **주의: 주기적 활성화 필요** |

**위험 시나리오:**
- `generated_codes` 테이블의 코드 데이터가 500MB를 초과
- 7일간 사용자가 없으면 DB 자동 일시정지

**대응 방안:**
```
DB 용량 관리:
├── generated_codes: 프로젝트당 최신 3버전만 보관
├── 90일 이상 미사용 프로젝트의 코드 자동 삭제
├── 코드 데이터 압축 저장 (gzip)
└── 월간 용량 모니터링 쿼리 실행

일시정지 방지:
├── UptimeRobot으로 5분마다 Health Check 호출
└── /api/health 엔드포인트 구현 (DB 조회 포함)
```

**용량 모니터링 쿼리:**
```sql
-- 테이블별 용량 확인
SELECT
  schemaname || '.' || tablename AS table_name,
  pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC;

-- generated_codes 테이블 행 수 및 평균 크기
SELECT
  COUNT(*) AS total_rows,
  pg_size_pretty(AVG(LENGTH(code_html) + LENGTH(code_css) + LENGTH(code_js))::bigint) AS avg_code_size,
  pg_size_pretty(pg_total_relation_size('generated_codes')) AS total_table_size
FROM generated_codes;
```

### 1.3 Gemini API (AI 코드 생성)

| 항목 | 무료 한도 (Gemini 1.5 Flash) | 주의사항 |
|------|------------------------------|---------|
| RPM (분당 요청) | 15 | 동시 사용자 제한 |
| RPD (일일 요청) | 1,500 | 하루 최대 생성 가능 수 |
| TPM (분당 토큰) | 1,000,000 | 충분 |
| TPD (일일 토큰) | 무제한 (Flash) | |

**위험 시나리오:**
- 동시 15명 이상 코드 생성 요청 시 429 에러
- 일일 1,500건 이상 생성 불가

**대응 방안:**
```
요청 관리:
├── 사용자당 일일 10회 생성 제한
├── 생성 요청 큐잉 (동시 요청 제한)
├── 429 에러 시 자동 재시도 (exponential backoff)
└── 유사 요청 캐싱 (동일 API+유사 컨텍스트)

폴백 전략:
├── 한도 초과 시 "잠시 후 다시 시도" 안내 + 예상 대기 시간
├── Gemini 1.5 Pro 교차 사용 (Pro도 무료 한도 있음)
└── Ollama 로컬 LLM 폴백 (서버 구축 시)

일일 한도 분배 (1,500 RPD 기준):
├── 사용자 생성: 1,200회 (사용자당 10회 × 120명)
├── 재생성: 200회
└── 시스템 예비: 100회
```

### 1.4 GitHub (소스 코드 + 생성 서비스 저장소)

| 항목 | 무료 한도 | 주의사항 |
|------|-----------|---------|
| 저장소 수 | 무제한 | |
| Actions 시간 | 2,000분 / 월 | CI/CD 파이프라인 |
| Packages Storage | 500 MB | 미사용 |
| API 요청 | 5,000 / 시간 (인증) | 서비스 배포 시 소비 |

**위험 시나리오:**
- 생성 서비스 배포마다 API 호출 3~5회 → 시간당 1,000~1,600건 배포 가능
- Actions 2,000분 = 약 33시간 빌드

**대응 방안:**
```
API 호출 관리:
├── 배포 API 호출 최소화 (batch 처리)
├── Rate limit 헤더 모니터링
└── 429 시 대기 후 재시도

Actions 시간 관리:
├── CI 최적화 (캐싱, 불필요한 단계 제거)
├── PR마다 전체 빌드 대신 변경분만 검사
└── 월간 사용량 추적
```

## 3. 한도 초과 시 긴급 대응 매뉴얼

### 3.1 Vercel 대역폭 초과
```
증상: 사이트 접근 불가 (429/503 에러)
긴급 대응:
1. Vercel Dashboard에서 사용량 확인
2. 다음 달 1일 자동 초기화까지 대기
   또는
3. 정적 페이지를 GitHub Pages로 임시 전환
4. 이미지/정적 파일 CDN으로 분리 검토
```

### 3.2 Supabase 용량 초과
```
증상: DB 쓰기 실패, 서비스 생성 불가
긴급 대응:
1. 오래된 generated_codes 삭제
   DELETE FROM generated_codes WHERE created_at < NOW() - INTERVAL '90 days';
2. 비활성 프로젝트 정리
3. 용량 재확인
```

### 3.3 Gemini API 한도 초과
```
증상: 코드 생성 요청 시 429 에러
긴급 대응:
1. RPM 초과: 1분 대기 후 자동 재시도
2. RPD 초과: "오늘 생성 한도에 도달했습니다" 안내
3. 대체 모델 사용 (Gemini Pro, 또는 다른 무료 API)
4. 다음 날 00:00 UTC 자동 초기화
```

### 3.4 Supabase 일시정지
```
증상: 모든 DB 연동 기능 실패
긴급 대응:
1. Supabase Dashboard → 프로젝트 → "Restore" 클릭 (수 분 소요)
2. UptimeRobot Health Check가 정상 설정되었는지 확인
3. 재발 방지: /api/health 엔드포인트 호출 주기 확인
```

## 4. 비용 발생 방지 안전장치

### 4.1 Vercel
- Hobby(무료) 플랜은 자동 과금 없음 (한도 초과 시 서비스 중단)
- Spend Management 설정 불필요 (무료 플랜)
- ⚠️ Pro 플랜으로 업그레이드 하지 않도록 주의

### 4.2 Supabase
- Free 플랜은 자동 과금 없음
- ⚠️ "Upgrade" 버튼 클릭 주의
- 결제 수단 등록하지 않음

### 4.3 Gemini API
- Free 티어는 과금 없음
- ⚠️ Billing 활성화하지 않음
- GCP 프로젝트에 Budget Alert 설정 ($0 알림)

### 4.4 GitHub
- Free 플랜에서 Actions 초과 시 자동 중단 (과금 없음)
- ⚠️ Billing settings에서 Spending limit: $0 확인

## 5. 사용량 예측 시뮬레이션

### 시나리오: 100명 DAU (Daily Active Users)

| 항목 | 계산 | 일일 | 월간 | 한도 | 여유 |
|------|------|------|------|------|------|
| 페이지 조회 | 100명 × 10페이지 × 500KB | 500MB | 15GB | 100GB | ✅ |
| DB 읽기 | 100명 × 20쿼리 × 2KB | 4MB | 120MB | 5GB | ✅ |
| 코드 생성 | 100명 × 2회 | 200회 | 6,000회 | 45,000회 | ✅ |
| Gemini API | 200회 | 200RPD | - | 1,500RPD | ✅ |
| DB 쓰기 | 200회 × 20KB(코드) | 4MB | 120MB | 500MB | ⚠️ |
| 배포 | 100회 | 100회 | 3,000프로젝트 | 200동시 | ⚠️ |

### 병목 지점
1. **Supabase DB 용량** (500MB): 약 4개월 후 한도 도달 → 정리 정책 필수
2. **Vercel 프로젝트 수** (200개): 약 2개월 후 한도 도달 → 비활성 정리 필수
3. **Gemini RPM** (15/분): 동시 15명 이상 생성 시 병목 → 큐잉 필수
